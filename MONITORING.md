# MONITORING — KPI и чек-поинты (саморевизия раз в 15–30 мин)

> Я в роли юзера: каждые **20 минут** сверяю фактические показатели с ожидаемыми. Источник данных:
> `session_log/<дата>.jsonl` (sessionlog) + статы панели (farm._stat: box_normal/stage/act,
> stages_ok/fail, merges, inv occupancy, records_ready) + игровой лог RECORDS.
> Цель: поймать деградацию РАНЬШЕ, чем она съест ночь. Любой 🔴 → диагностика + фикс.

## Базовые инварианты (должны держаться ВСЕГДА)
| KPI | 🟢 норма | 🟡 насторожиться | 🔴 чинить |
|---|---|---|---|
| `records_ready` (лог развёрнут) | true | мигает | false ≥2 проверки → пере-развернуть (clear_panels→pin_and_expand) |
| Оба лога (бот+игра) движутся | обновление <5 мин | 5–20 мин тишина | ≥20 мин стоят оба = **ХЭНГ** (паттерн tbh-bot-hang) → рестарт батником |
| Панель жива (heartbeat) | да | — | нет → процесс умер, рестарт |
| Инвентарь occupancy | колеблется, падает после save | стабильно высоко | =макс несколько циклов → save/sort не срабатывает |
| defeats / stage_fail за окно | 0 | 1–2 | ≥3 на одной стадии → **откат на N уровней** (pullback) |
| Игра foreground при кликах | да | — | нет (юзер активен) — норм ночью, но клики пропускаются |

## Продуктивность (сверять с ожиданием по механике)
КД: обычный сундук 5 мин/уровень, боссовый 7 мин/уровень (community, проверять замером).
| KPI | ожидание за 20 мин (ориентир) | 🔴 |
|---|---|---|
| Новые сундуки (box_normal+stage+act) | растёт стабильно; при хопе ≥ одностадийного | 0 за 20 мин при живой игре → лог-детект сундука сломан |
| `stages_ok` (зачистки/runs) | растёт (зависит от clear-time стадии) | 0 при aggressive+живой игре → бой не идёт / стадия не вывозится |
| Почта | собирается раз в ~5.5 мин (mail_every_sec 330) | не собирается ≥15 мин → mail-путь сломан |
| Мерж | 0 (куб OFF по умолч.) | >0 при выключенном кубе = БАГ безопасности |

## Чек-поинт (что делаю каждые 20 мин)
1. Прочитать хвост `session_log/<сегодня>.jsonl` → когда последнее событие? (тишина = хэнг-флаг).
2. Снять `box_*`, `stages_ok/fail`, `inv occupancy`, `records_ready` → записать в NIGHT_LOG.
3. Сравнить дельту с прошлым чек-поинтом (растут ли сундуки/зачистки? не копится ли инвентарь?).
4. defeats/fail ≥3 на стадии → инициировать pullback (откат на N ур.).
5. records_ready=false → пере-развернуть лог.
6. Хэнг (оба лога ≥20 мин стоят) → НЕ рестартить игру (только с юзером); рестарт панели батником — можно.
7. Записать строку-итог в NIGHT_LOG.md (время, дельты, вердикт 🟢/🟡/🔴, что сделал).

## Ожидаемое «здоровое» окно (эталон для сверки)
- За 20 мин при живом aggressive-фарме на проходимой стадии: stages_ok ↑ на несколько, сундуки ↑,
  инвентарь не уперся в макс (был save), defeats=0, лог ready, почта собралась хотя бы раз за ~5.5 мин.
- Если всё это так → 🟢, ничего не трогаю. Любое отклонение → пункт-фикс выше.
